# Google Sheets ⇄ Supabase Sync (Apps Script)

สคริปต์นี้ทำให้ Google Sheets เป็น **ข้อมูลสำรอง (backup)** และเป็น **ที่แก้ไขข้อมูลสินค้า** ควบคู่กับ Supabase:

- ⏰ ซิงก์อัตโนมัติทุก 30 นาที (ปรับได้)
- 🆕 สร้างรายการสั่งซื้อใหม่ในแอดมิน → สร้าง Google Spreadsheet ใหม่ให้อัตโนมัติ (แท็บ `Products` + `Orders`)
- ✏️ แก้ไขสินค้าในชีต (ราคา มัดจำ สต็อก ชื่อ ฯลฯ) → ค่าจะไหลเข้า Supabase ในการซิงก์รอบถัดไป
- 🧾 ออเดอร์จากหน้าเว็บ → ถูกสำรองลงแท็บ `Orders` ของชีตแต่ละรายการ
- 🗂️ ไฟล์ **"Order Hub Backup"** เก็บสำเนาตาราง `order_lists` และคลังสินค้า (`stock_items`) ทั้งหมด

## กติกาการซิงก์ (สำคัญ — อ่านก่อนใช้)

| ข้อมูล | ทิศทาง | หมายเหตุ |
|---|---|---|
| สินค้า (แท็บ Products) | **สองทาง** | เซลล์ที่คุณแก้ในชีต "ชนะ" — ส่วนค่าที่ไม่ได้แก้ (เช่นสต็อกที่ถูกหักจากออเดอร์) ใช้ค่าจาก Supabase |
| ออเดอร์ (แท็บ Orders) | Supabase → ชีต | สำรองอย่างเดียว การแก้ในแท็บนี้จะถูกเขียนทับ |
| รายการสั่งซื้อ + คลังสินค้า (ไฟล์ Backup) | Supabase → ชีต | สำรองอย่างเดียว |
| การลบแถว | ❌ ไม่ซิงก์ | ลบแถวในชีตไม่ได้ลบใน Supabase (แถวจะกลับมาเอง) — ให้ลบผ่านหน้าแอดมิน |

หลักการตรวจว่า "เซลล์ไหนถูกแก้ในชีต": สคริปต์เก็บสำเนาที่เขียนล่าสุดไว้ในแท็บซ่อน `_snapshot`
เซลล์ที่ต่างจาก snapshot = คุณแก้เอง → ส่งเข้า Supabase แล้วเขียน snapshot ใหม่
**อย่าลบ/แก้แท็บ `_snapshot`** ไม่งั้นรอบถัดไปจะกลายเป็น baseline ใหม่ (ชีตถูกเขียนทับจาก Supabase หนึ่งรอบ)

> ⚠️ **การซิงก์ครั้งแรก** ของชีตแต่ละไฟล์เป็น "baseline" — เขียนข้อมูลจาก Supabase ลงชีตอย่างเดียว
> การแก้ไขในชีตจะเริ่มไหลเข้า Supabase ตั้งแต่รอบที่สองเป็นต้นไป
> ถ้าชีตเก่ามีข้อมูลใหม่กว่า Supabase ให้กดปุ่ม "ซิงก์จากชีต" ในหน้าแอดมินก่อนเปิดใช้ระบบนี้

## วิธีติดตั้ง (~10 นาที)

### 1. สร้างโปรเจกต์ Apps Script
1. ไปที่ https://script.new (ล็อกอินบัญชี Google ที่ต้องการให้เป็นเจ้าของชีต)
2. ลบโค้ดตัวอย่าง แล้ววางเนื้อหาไฟล์ `apps-script/sync.gs` ทั้งหมด → บันทึก (ตั้งชื่อ เช่น "Order Hub Sync")

### 2. ตั้งค่า Script Properties
ไปที่ **Project Settings (⚙️) → Script Properties → Add script property** แล้วเพิ่ม:

| Property | ค่า |
|---|---|
| `SUPABASE_URL` | เช่น `https://xxxx.supabase.co` |
| `SUPABASE_SERVICE_KEY` | **service_role / secret key** จาก Supabase → Settings → API Keys (ห้ามเอาไปใส่ใน config.js ของเว็บเด็ดขาด — ที่นี่ปลอดภัยเพราะอยู่ฝั่งเซิร์ฟเวอร์) |
| `SHARED_SECRET` | รหัสยาว ๆ ที่ตั้งเอง เช่นสุ่ม 32 ตัวอักษร — ต้องตรงกับ `SYNC_WEBAPP_SECRET` ใน `js/config.js` |
| `DRIVE_FOLDER_ID` | *(ไม่บังคับ)* ID โฟลเดอร์ใน Drive สำหรับเก็บชีตที่สร้างใหม่ |

### 3. เปิดใช้ตัวตั้งเวลา (trigger)
1. ในตัวแก้ไข เลือกฟังก์ชัน `setupTrigger` แล้วกด **Run**
2. อนุญาตสิทธิ์ที่ระบบขอ (Sheets, Drive, external requests)
3. เสร็จแล้วจะซิงก์อัตโนมัติทุก 30 นาที (แก้ค่า `SYNC_EVERY_MINUTES` ในโค้ดได้: 1/5/10/15/30)

ทดสอบได้โดยเลือกฟังก์ชัน `testSyncOnce` แล้ว Run → ดูผลใน **Executions / Logs**

### 4. Deploy เป็น Web App (ให้หน้าแอดมินเรียกได้)
1. **Deploy → New deployment → Web app**
2. ตั้งค่า: *Execute as* = **Me**, *Who has access* = **Anyone**
3. กด Deploy แล้วคัดลอก **Web app URL** (ลงท้าย `/exec`)

### 5. ใส่ค่าใน `js/config.js` ของเว็บ
```js
window.SYNC_WEBAPP_URL    = 'https://script.google.com/macros/s/XXXXX/exec';
window.SYNC_WEBAPP_SECRET = 'รหัสเดียวกับ SHARED_SECRET';
```

เสร็จแล้วในหน้าแอดมิน (แท็บภาพรวม) จะมีปุ่ม **🔄 ซิงก์ชีต** และการสร้างรายการใหม่จะสร้าง Google Sheet ให้ทันที

## คำถามที่พบบ่อย

**ต้องแชร์ชีตให้ใครไหม?** — ไม่ต้อง ชีตที่สร้างใหม่เป็นของบัญชี Google ที่รันสคริปต์
แต่ชีตเก่าที่เคยนำเข้า (source spreadsheet เดิม) ต้องแชร์ให้บัญชีนี้เปิดได้ ไม่งั้นรายการนั้นจะถูกข้าม (ดู error ใน log)

**แก้ไขพร้อมกันทั้งชีตและแอปได้ไหม?** — ได้ ตราบใดที่ไม่แก้ "เซลล์เดียวกัน" ระหว่างรอบซิงก์
ถ้าแก้เซลล์เดียวกันทั้งสองฝั่ง ค่าในชีตชนะ

**SHARED_SECRET ปลอดภัยแค่ไหน?** — ค่านี้อยู่ใน `config.js` ซึ่งเปิดดูได้จากซอร์สหน้าเว็บ
มีไว้กันคนแปลกหน้าสั่งซิงก์เล่นเท่านั้น ข้อมูลจริงถูกป้องกันด้วย service key ที่อยู่ใน Script Properties ฝั่ง Google เสมอ

**เปลี่ยนความถี่ซิงก์?** — แก้ `SYNC_EVERY_MINUTES` บนหัวไฟล์ แล้ว Run `setupTrigger` ใหม่หนึ่งครั้ง
